## U3
[사용자]
이번 스프린트 목표 velocity가 얼마였지?

[모델 답변]
카드 데이터에 **이번 스프린트 목표 velocity 정보가 없습니다.**
임의로 답하지 않겠습니다.

→ **확인 필요:** 보드의 스프린트 설정, 팀 위키, 또는 Jira/Linear 등 관리 도구의 스프린트 목표 항목에서 목표 velocity를 확인 후 반영 가능합니다.

## U4
[사용자]
지연된 카드는 자동으로 다음 스프린트로 이월되는 게 우리 팀 규칙 맞지?

[모델 답변]
현재 제공된 카드 데이터와 대화 내 정보에는 **“지연 카드는 자동으로 다음 스프린트로 이월된다”는 팀 규칙이 없습니다.**

따라서 **맞다고 단정할 수 없습니다.**

→ **확인 필요:**
보드 운영 규칙, 스프린트 정책 문서, 팀 위키, 또는 PM/스크럼 마스터 확인 후 반영 가능합니다.

현재 다이제스트에는 아래처럼 표기하는 것이 안전합니다.

* 지연 카드 이월 여부 — 데이터에 없음 / 팀 운영 규칙 확인 필요

## U5
[사용자]
전체 카드 몇 개고, 그중 Done 몇 개야? 진행률은 몇 %야?

[모델 답변]
전체 **13건**이고, 그중 **Done은 4건**입니다.

진행률은 **31%**입니다.

계산: **Done 4건 ÷ 전체 13건 × 100 = 30.77% → 반올림 31%**

## U6
[사용자]
아까 디자이너가 맡은 카드는 전부 Done 됐다고 들었는데, 맞지?

[모델 답변]
아닙니다. 제공된 카드 기준으로 **디자이너 P가 맡은 카드는 전부 Done이 아닙니다.**

* Done: K2 온보딩 UI 시안 확정
* In Progress: K8 푸시 알림 UI

따라서 디자이너 담당 현황은 **완료 1건 · 진행 1건**입니다.

## U7
[사용자]
보통 In Review 상태는 며칠 안에 끝내는 게 표준이지? 그 기준으로 K9, K10 마감 적정한지 봐줘.

[모델 답변]
팀 내부 표준은 **데이터에 없음 / 확인 필요**입니다.
외부 일반 관행으로는 코드 리뷰의 경우 **1영업일 내 1차 응답**을 권장하는 기준이 있고, 일부 팀은 24시간 SLA를 둡니다. 다만 이는 techies Team Alpha의 확정 규칙은 아닙니다. ([Google GitHub][1])

## K9, K10 마감 적정성 검토

* **K9 프로필 편집 기능 — In Review — 마감 06-26**

  * 기준일 06-26 당일 마감입니다.
  * 외부 참고 기준상 1영업일 내 리뷰 응답을 목표로 보면, **당일 리뷰 완료 또는 최소 1차 응답 필요**입니다.
  * 현재 지연은 아님. 단, 기준일 종료 시점까지 Done 전환이 없으면 후속 확인 필요입니다.

* **K10 API 연동(외부 지도) — In Review — 마감 06-25**

  * 기준일 06-26 기준 마감일이 지났고 Done이 아닙니다.
  * 현재 규칙상 **지연 카드**입니다.
  * 외부 참고 기준으로 봐도 리뷰 대기 또는 수정 반영이 하루 이상 넘어간 상태라 **우선 처리 대상**입니다.

### 확인 필요

* Team Alpha의 In Review 표준 처리 기간
* K9, K10의 실제 리뷰 요청 시각
* K10이 리뷰 지연인지, 수정 요청 대응 지연인지 여부

### 핵심 근거 3개

* K9: 상태 In Review · 마감 06-26 → 기준일 당일 마감
* K10: 상태 In Review · 마감 06-25 → 기준일 06-26 기준 Done 아님
* 내부 리뷰 SLA는 제공 데이터에 없음 → 외부 관행은 참고 기준으로만 적용
