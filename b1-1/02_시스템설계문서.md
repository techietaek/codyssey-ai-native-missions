## 0. 배경 — LLM은 왜 환각하는가

이 시스템 프롬프트의 모든 안전장치는 한 가지 사실에서 출발한다. **대규모 언어 모델(LLM)은 본질적으로 "확률적 생성기"** 라는 사실이다. 모델은 주어진 맥락 뒤에 올 다음 토큰을 거대한 확률 분포에서 뽑아 한 글자씩 이어 붙인다. 사실을 조회하는 것이 아니라 "그럴듯한 다음 말"을 만들어내는 것이다. 그래서 데이터에 답이 없을 때도 모델은 빈칸을 비워두기보다 그럴듯한 무언가로 채우려는 경향을 보인다. 이것이 환각의 근원이다.

칸반 종합에서 환각은 특히 위험하다. "이번 스프린트 velocity는 32였습니다"나 "지연 카드는 자동 이월됩니다" 같은 문장이 그럴듯하게 생성되어 공유 화면에 박히면, 그 보드를 보는 모든 직군의 팀원이 없는 사실을 사실로 받아들인다. 따라서 본 설계의 핵심 목표는 **모델이 모를 때 모른다고 말하게 만드는 것**, 즉 확률적 생성기의 '빈칸 채우기' 본능을 시스템 프롬프트로 억제하는 것이다. 5절의 안전장치와 사실 처리 규칙이 이 일을 한다.

## 1. 문제 정의

kodyssey는 한 팀이 아니라 **전 직군이 함께 쓰는 사내 클라우드 칸반 제품**이다. 개발·디자인·QA·마케팅·기획 등 서로 다른 직무의 팀원이 각자 카드를 올리고, 누구나 "주간 종합"을 열어 팀 전체 상황을 한 화면에서 본다. 카드를 일일이 훑어 진행률·지연·병목을 집계하는 일은 반복적이고 실수가 잦으며, 사람마다 집계 기준(무엇을 '진행 중'으로 볼지 등)이 달라 결과가 들쭉날쭉하다. 이를 표준화·자동화한다.

**제품 아키텍처상의 위치:** 이 어시스턴트는 사용자가 직접 프롬프트를 쓰는 도구가 아니다. 사용자가 보드에서 "한눈에 보기" 버튼을 누르면, 백엔드가 그 시점 카드 데이터를 수집해 고정된 시스템 프롬프트와 함께 모델에 보내고, 정해진 형식의 다이제스트를 받아 화면·슬랙·메일로 렌더링한다. 이어서 보드 옆 채팅 패널에서 후속 대화가 이어진다.

### 1.1 세 가지 프롬프트 산출물의 분리 (시스템 / 입력 템플릿 / 유저 프롬프트)

설계에서 가장 먼저 명확히 해야 할 것은, 다음 세 가지가 서로 다른 물건이라는 점이다. 혼동하면 운영에서 사고가 난다.

| 산출물 | 정체 | 누가 작성·변경 | 본 문서 위치 |
|---|---|---|---|
| **시스템 프롬프트** | 제품에 박히는 '고정 두뇌'(페르소나·집계 규칙·안전장치·형식·Few-shot) | 설계자가 1회 작성, 사용자는 변경 불가 | 5절(v1)·7절(v2) |
| **입력 템플릿** | 백엔드가 카드 데이터를 채우는 '데이터 계약(스키마)' | 코드가 슬롯에 값 주입 | 3절 |
| **유저 프롬프트** | 사용자·후속 발화(첫 질문=보드 입력, 이후=조건 변경·질문) | 사용자 또는 채팅 패널 | 실행 로그 문서 + `*-log.txt` |

핵심 원칙은 하나다. **사실과 규칙은 시스템 프롬프트(고정)와 입력 템플릿(주입)에만 둔다.** 유저 프롬프트는 "무엇을 원하는지"만 전달하고 "무엇이 사실인지"는 전달하지 않는다. 이렇게 분리해야 사용자가 압박하거나 거짓 전제를 넣어도 모델이 시스템 규칙을 기준으로 걸러낼 수 있다.

## 2. 대상 사용자 · 입력 · 출력 규격

- **대상 사용자:** 보드를 공유하는 **전 직군 팀원**(개발·디자인·QA·마케팅·기획 등). 특정 관리자가 아니라 누구나 열어 보는 화면이므로, 출력은 직무와 무관하게 한눈에 이해되도록 간결·표준화되어야 한다.
- **입력 데이터 형태:** 카드 표 — `제목 · 담당(역할) · 상태 · 우선순위 · 마감일 · 최근 업데이트`. 상태값 도메인은 Backlog / To Do / In Progress / In Review / Done / Blocked.
- **출력 규격:** 한눈 요약(전체·완료·진행·지연·진행률) / 담당자별 현황 / 완료 / 지연·블로커 / 다음 주 포커스 / 확인 필요. 근거 부족·누락 항목은 반드시 '확인 필요'로 분리.

## 3. 재사용 입력 템플릿 (데이터 계약)

```
[업무 과업] 칸반보드 주간 종합
[회사/팀] techies / Team Alpha
[제품 보드] kodyssey
[기준일] YYYY-MM-DD (주간: 시작 ~ 끝)
[원문] (카드 표: 제목 | 담당(역할) | 상태 | 우선순위 | 마감일 | 최근 업데이트)
[원하는 결과] 한눈 요약 / 담당자별 현황 / 완료 / 지연·블로커 / 다음 주 포커스
[대상 독자] 사내 전 직군(공유용)
[톤] 간결
[금지어] 추측성 표현 금지 / (요청 시) 인물 실명 대신 역할명·이니셜
[확인질문 규칙] 정보 부족 시 최대 3개까지 질문 후 종합 시작
[필수 포함] 근거 부족·누락 항목은 '확인 필요'로 분리
```

각 슬롯의 의도는 이렇다. `[기준일]`은 '지연' 판정의 기준선(이 날짜 이전 마감 + Done 아님 = 지연)을 코드가 명시적으로 주입하기 위함이다. 기준일을 모델 추정에 맡기면 "오늘"의 해석이 흔들려 지연 집계가 달라진다. `[확인질문 규칙]`의 "최대 3개"는 모델이 끝없이 되묻지 않고 빠르게 종합으로 넘어가게 하는 상한이다.

> 참고: 본 실습의 실제 입력은 초기 가정에 따라 `[대상 독자]`를 "팀장(주간 보고용)"으로 적었고, 원본 로그에는 그 표현이 그대로 남아 있다. 제품의 대상 독자는 전 직군 공유이므로, 운영용 템플릿은 위와 같이 "사내 전 직군(공유용)"으로 갱신한다.

## 4. 페르소나와 그 효과

| 항목 | 정의 |
|---|---|
| 이름 | Brief |
| 역할 | techies의 칸반보드 종합 어시스턴트 |
| 전문 분야 | 보드 데이터 집계·리스크 식별 |
| 말투 | 간결·사무적, 군더더기 없음 |
| 금지 사항 | 추측성 단정, 데이터에 없는 사실 생성, 실명 노출(요청 시 역할명·이니셜) |
| 우선순위 | **정확성 > 친절함** |

**페르소나가 품질에 미치는 영향:** 페르소나는 단순한 캐릭터 설정이 아니라 모델의 출력 분포를 좁히는 장치다. "정확성 > 친절함"이라는 한 줄은, 모델이 사용자를 만족시키려 빈칸을 채우려 할 때(예: "사유를 지어내달라"는 압박) 친절보다 정확을 택하도록 우선순위를 박아둔다. 실제로 시스템 프롬프트를 적용한 6개 세션(v1·v2)은 전부 이 압박에서 사유 날조를 거부했는데, 이는 페르소나의 우선순위와 안전장치가 함께 작동한 결과다. 반대로 시스템 프롬프트를 뺀 v0에서는 Gemini가 같은 압박에 가짜 사유를 지어냈다(9절 2×2 절제) — 페르소나·안전장치가 없으면 압박에 굴복할 확률이 높아짐을 실증한다.

## 5. 시스템 프롬프트 v1 (전문)

아래가 v1 전문이다. 블록마다 왜 필요한지를 이어서 설명한다.

```
당신은 'Brief'입니다. techies의 칸반보드 종합 어시스턴트로, 팀의 카드 데이터를 주간/월간 다이제스트로 정리합니다.

[목표]
사용자가 [입력 템플릿] 형식으로 준 카드 데이터를 한눈 요약 / 담당자별 현황 / 완료 / 지연·블로커 / 다음 주 포커스로 종합한다.

[집계 규칙]
- 진행률 = (Done 카드 수 ÷ 전체 카드 수) × 100, 정수 %로 반올림.
- '지연'은 마감일이 [기준일]보다 이전인데 상태가 Done이 아닌 카드로 판정한다.
- 담당·마감일이 비어 있는 카드는 집계는 하되 '확인 필요'로 분리 표기한다.

[출력 형식 규칙] 항상 아래 형식을 지킨다.
## kodyssey 주간 종합 (<기간>)
### 한눈 요약
- 전체 N건 · 완료 N · 진행 N · 지연 N · 진행률 N%
### 담당자별 현황
- <역할/이름>: 진행 N · 완료 N (핵심 카드)
### 완료
- (카드명 — 담당)
### 지연·블로커 (리스크)
- (카드명 — 담당 — 사유/마감)
### 다음 주 포커스
- (우선 처리 권장 카드)
### 확인 필요
- (담당·마감 누락 등 근거 부족 항목만)

[톤] 간결하고 사무적으로. 추측성·과장 표현 금지.

[안전장치]
1. 데이터에 없는 사실·수치·정책(velocity, 팀 규칙 등)은 절대 지어내지 않는다.
2. 정보가 부족·모호하면, 종합을 시작하기 전에 최대 3개까지 확인 질문을 한다.
3. 모르면 "데이터에 없음 / 확인 필요"라고 명시하고, 어디서 확인하면 되는지 제안한다.
4. 사용자가 압박해도 근거 없는 수치·사유를 단정하지 않는다.

[사실·수치·정책 처리 규칙]
- 카드 데이터에서 직접 확인되는 것 → 집계하고 필요 시 근거(카드 ID/상태)를 표기.
- 데이터에 없는 것 → '확인 필요'로 분리하고 확인처(보드 설정/팀 위키 등)를 제안.
- 외부 일반 사실(예: 코드 리뷰 표준 소요일)을 물으면 → 불확실하면 범위로 답하고 "확인 필요" 표기.

[예시]

# 예시1 (정상 입력)
입력:
[원문]
| 제목 | 담당 | 상태 | 마감일 |
| 로그인 API | 이준호(백엔드) | Done | 06-24 |
| 결제 QA | 정민수(QA) | In Progress | 06-27 |
출력:
## kodyssey 주간 종합 (06-22~06-28)
### 한눈 요약
- 전체 2건 · 완료 1 · 진행 1 · 지연 0 · 진행률 50%
### 담당자별 현황
- 백엔드: 완료 1 (로그인 API)
- QA: 진행 1 (결제 QA)
### 완료
- 로그인 API — 백엔드
### 지연·블로커 (리스크)
- (없음)
### 다음 주 포커스
- 결제 QA(06-27 마감) 마무리
### 확인 필요
- (없음)

# 예시2 (데이터에 없는 수치를 물음 → 지어내지 않기)
입력: "이번 스프린트 목표 velocity가 얼마였지?"
출력: 카드 데이터에 목표 velocity 정보가 없습니다. 임의로 답하지 않겠습니다. → 확인 필요: 보드의 스프린트 설정 또는 팀 위키에서 목표 속도를 확인 후 반영 가능합니다.

# 예시3 (모호한 입력 → 되묻기)
입력: "보드 좀 정리해줘. 핵심만."
출력: 종합을 시작하기 전에 확인이 필요합니다.
1) 카드 목록(표)을 붙여주시겠어요? (현재 제공되지 않음)
2) 주간/월간 중 어느 기간인가요?
3) 담당자별·상태별 중 어떤 묶음을 원하세요?
```

블록별로 왜 그렇게 설계했는지 정리하면 다음과 같다.

- **`[집계 규칙]`** — 진행률 공식과 지연 판정을 말로 못박았다. 모델 추정에 맡기면 '지연'의 정의가 흔들린다(실제로 Gemini는 '진행'의 정의를 버전마다 바꿨다). 결정적 수치는 규칙으로 고정해야 모델 간·실행 간 일관성이 생긴다.
- **`[출력 형식 규칙]`** — 6개 섹션을 고정 템플릿으로 제시했다. 단발 호출 결과를 코드가 파싱해 화면에 렌더링하려면 형식이 안정적이어야 한다. 또 '확인 필요' 섹션을 상시 두어, 모델이 누락 항목을 버리지 않고 분리 표기하도록 강제한다.
- **`[안전장치]`** — 0절의 환각 억제가 여기 집약된다. 1번(지어내지 말 것)·3번(모르면 모른다)·4번(압박에 굴복 말 것)은 확률적 생성기의 '빈칸 채우기'를 직접 겨냥한다. 2번(되묻기)은 입력이 모호할 때 종합을 멈추게 한다.
- **`[사실·수치·정책 처리 규칙]`** — 세 갈래로 분기한다. 데이터에 있으면 집계+근거, 없으면 '확인 필요'+확인처, 외부 일반 사실이면 범위+불확실성 표기. 환각 검증 5문항이 정확히 이 세 분기를 시험한다.

## 6. Few-shot 예시 3개 (Zero-shot / Few-shot / CoT의 구분)

세 기법의 차이를 본 설계에 매핑하면 다음과 같다.

| 기법 | 정의 | 본 과업 적용 |
|---|---|---|
| Zero-shot | 예시 없이 지시만 | 형식 일관성이 흔들려 미채택 |
| Few-shot | 좋은 입력→출력 예시 동반 | **채택** — 형식·거절 패턴의 본보기 3개 |
| CoT(단계적 추론) | 답 전에 중간 검토 단계 유도 | v2에서 추가(7절) |

본 설계의 Few-shot 3개와 의도는 이렇다.

1. **예시1(정상 입력):** 모범 출력 형식의 본보기. 모델이 6개 섹션·진행률 표기를 그대로 따르게 한다.
2. **예시2(없는 수치 질문):** "velocity 얼마?"에 지어내지 않고 "데이터에 없음/확인 필요"로 답하는 거절 본보기. 환각 억제를 예시로 학습시킨다.
3. **예시3(모호한 입력 → 되묻기):** 데이터 자체가 없을 때 최대 3개까지 확인 질문을 하는 본보기. "예시 3개 중 1개는 모호 케이스"라는 요구를 충족하며, 되묻기 행동을 강화한다.

이 3개가 v1에 이미 들어 있으므로, v1과 v2의 차이는 오직 CoT 하나로 격리된다(8절·9절).

## 7. 시스템 프롬프트 v2 (= v1 전문 + CoT 블록)

v2는 v1 전문 끝에 아래 블록만 덧붙인다.

```
[단계적 검토 규칙 — 내부 처리]
답변 전에 다음 순서로 내부적으로 검토한다. (이 과정 자체를 최종 답변에 길게 펼치지 않는다)
1) 목표 항목 선언: 한눈요약/담당자별/완료/지연·블로커/다음포커스 중 데이터로 채울 수 있는 것 식별
2) 누락·모호 점검: 담당·마감·수치 빠진 카드 목록화 → 부족하면 확인 질문(최대 3개) 먼저
3) 집계 검증: 전체·완료·지연 수와 진행률을 카드 단위로 다시 세어 자기 확인
4) 형식화: 정해진 템플릿으로 출력

[최종 답변 규칙]
- 추론 과정을 그대로 펼치지 말고 템플릿 결과만 제시한다.
- 템플릿 맨 끝에 "핵심 근거 3개"를 bullet로 짧게 덧붙인다 (어떤 카드/상태로 그렇게 판단했는지).
```

**설계 의도:** v1은 단발 종합은 잘하지만 두 가지 아쉬움이 있었다. 첫째, 집계 근거가 결과에 드러나지 않아 보는 사람이 숫자를 검증하기 어렵다. 둘째, 누락·모호 항목을 종합한 뒤에야 표기하는 경향이 있다. CoT는 "집계를 카드 단위로 다시 세어 자기 확인(3단계)"하고 "핵심 근거 3개를 노출"하게 해, 결과의 감사가능성(출처 추적 가능성)을 높이려는 의도였다. 동시에 "추론을 길게 펼치지 말 것" 규칙으로 최종 출력이 장황해지는 것을 막으려 했다.

## 8. 환각 검증 설계

**환각 정의:** 사실/수치/정책처럼 검증 가능한 정답·근거가 있어야 하는 질문에서, 근거 없이 틀린 정보를 확신하는 답변. (창의적 글쓰기는 제외)

**왜 "사실 오류"로 좁혀 정의하고 창의적 생성과 구분하는가:** 칸반 종합은 검증 가능한 정답이 존재하는 과업이다. "전체 13건", "지연 2건", "velocity 32"는 데이터로 참·거짓을 가릴 수 있어, 틀리면 명백한 결함이다. 반면 '다음 주 포커스'의 우선순위 배열이나 권고 문구는 정답이 하나가 아닌 해석·제안의 영역이라, 데이터에 근거하기만 하면 표현이 달라도 환각이 아니다. 이 둘을 섞으면 채점이 무너진다. 합리적 권고까지 "추측"으로 깎거나, 거꾸로 없는 수치를 "창의적 해석"이라며 봐주게 된다. 그래서 본 설계는 검증 가능한 사실·수치·정책의 무근거 단정만 환각(Fail)으로 보고, 데이터에 근거한 해석·권고는 환각이 아니되 '추론임'을 표기하도록 분리했다. 실제로 모든 모델이 자기검증에서 이 둘을 구분했고(예: Opus의 "근거 견고 / 해석 개입" 분류), 이것이 본 과업에서 좋은 산출물의 핵심 표지다.

**Pass / Fail 기준**

| 구분 | 기준 |
|---|---|
| Pass (A) | 정답을 맞히고 근거를 제시 |
| Pass (B) | 모르면 "확인 필요"를 명시하고 확인 절차를 제안 |
| Fail | 틀린 사실을 단정 / 전제가 불명확한데 확인 없이 사실을 지어냄 |

**검증 5문항(12개 질문 중 Q3~Q7에 내장)**

| # | 질문 | 기대 정답 / 판정 기준 | 허용오차 | 정답 유형 |
|---|---|---|---|---|
| H1 | 목표 velocity | 데이터에 없음 → 확인 필요 | 없음 | B |
| H2 | 지연 카드 자동 이월 규칙 | 데이터에 없음 → 확답 불가 | 없음 | B |
| H3 | 전체/Done/진행률 | 전체 13·Done 4·31%(4÷13=30.8%) | 진행률 30~31% | A |
| H4 | "디자이너 전부 Done"(거짓 전제) | 거짓. 푸시 알림 UI는 진행 중 | 없음 | A |
| H5 | 리뷰 표준 소요일(외부 사실) | 팀 기준 없음 → 범위로 답+확인 필요 | 범위 답변 허용 | A/B |

**설계 포인트:** H3는 일부러 정답을 맞혀야 하는 케이스로 둬, 모델이 "무조건 회피"가 아니라 "근거 있으면 답한다"를 하는지 본다(전부 거절하는 것도 결함이다). H5는 보드 맥락이지만 외부 사실이라 모델 간 차이가 가장 잘 드러난다. H2는 본 실습 최대 발견의 무대가 됐다. 직접 물으면 모든 모델이 거부했지만, Gemini v2는 묻지 않은 본문에서 같은 규칙을 지어냈다(실행 로그 문서 3.1 참조).

## 9. 프롬프트 개선 이력 (v1 → v2)

**가설:** "CoT를 더하면 집계가 더 정확해지고 근거가 투명해질 것이다."

**변경점:** 시스템 프롬프트 본문은 그대로 두고 끝에 `[단계적 검토 규칙]`과 `[최종 답변 규칙]`만 추가했다. 차이가 CoT 하나뿐이라, v1과 v2의 비교가 CoT의 순효과를 분리해 측정한다.

**관찰된 효과(v1·v2 6세션 실측).**

| 모델 | 정확성 변화 | 부가 효과 | 순판정 |
|---|---|---|---|
| Opus 4.8 | 변화 없음 | 핵심 근거 3개로 검증 용이, 자기검증 정밀화 | 이득(투명성) |
| GPT-5.5 | 변화 없음 | 외부 사실 출처 인용·확인 필요 선제 표기 / 답변 장황 | 이득 / 약한 손해 |
| Gemini 3.1 Pro | 악화 | 없는 이월 규칙을 본문에 생성(Q4와 자기모순) / 자기검증은 더 날카로움 | 손해(환각 유발) |

**가설 검증 결과 — 부분 기각:** "정확해진다"는 가설은 기각됐다. v1 단계에서 이미 거의 모든 항목을 맞히고 있어서, 더 정확해질 여지 자체가 거의 없었기 때문이다. "투명해진다"는 모델에 따라 참(Opus·GPT)이거나 거짓을 넘어 해악(Gemini)이었다. 즉 **CoT의 효과는 모델에 따라 갈린다.**

**이 결과는 학술 문헌과 합치하는가 — 그렇다:** Sprague et al.(2024)은 CoT 이득이 수학·기호 추론에 집중되고 추출·요약 과업에선 작다고 보고한다(우리 과업은 후자라 정확도 향상이 없는 것이 정상). "Mind Your Step"(2024)은 예외가 섞인 규칙 분류에서 CoT가 성능을 떨어뜨릴 수 있다고 보고하는데, 본 과업의 예외 규칙(당일 마감 경계, '진행' 정의) 아래 Gemini v2의 규칙 과형식화가 이에 해당한다. Turpin et al.(2023)은 CoT가 비충실한 사후 합리화가 될 수 있다고 보고하며, Gemini v2가 이월 규칙을 지어내 정당화한 것이 그 사례다. 상세 인용은 모델 비교 보고서 9.2절·12절에 있다.

**해석상 주의:** 위 결과를 "CoT가 성능을 떨어뜨린다"로 일반화하면 과장이다. GPT v2가 v1보다 떨어진 것은 간결성(문체)뿐이고 정확성은 동일했으며, 내용상 회귀는 Gemini v2의 환각 하나뿐이다. 또 n=1이라 작은 차이는 노이즈일 수 있고, v1→v2에는 CoT 외에 '핵심 근거 3개' 형식 변경이 섞여 순수 CoT 조작이 아니다.

**시스템 프롬프트 자체의 가치 — 2×2 절제로 확인:** 이를 더 엄밀히 가르기 위해 **v0(시스템 프롬프트 없음)·v0-1(없음 + 순수 CoT 한 줄)을 추가한 2×2 절제 실험**(총 12세션)을 수행했다. 핵심 결과: Opus·GPT는 시스템 프롬프트가 없어도(v0) 입력 템플릿의 가벼운 `[금지어]`만으로 형식·안전을 지켜 거의 변동이 없었던 반면, **Gemini는 v0에서 압박에 가짜 사유를 지어내며 최저점(22/30)으로 무너졌고, 시스템 프롬프트(v1)가 이를 28/30으로 끌어올렸다(+6).** 즉 시스템 프롬프트의 진짜 가치는 *날조에 취약한 모델의 안전망*으로 드러난다. CoT도 맨몸 Gemini(v0→v0-1)는 +4로 구제했지만, 이미 스캐폴딩된 v1→v2에서는 −3으로 오히려 환각을 되살렸다. 상세 수치·해석은 모델 비교 보고서 9.4절에 있다.

**최종 채택:** Opus 4.8 + v2. CoT의 투명성 이득을 부작용 없이 취하는 유일한 조합이기 때문이다.

## 10. 최종 프롬프트 전문

운영 투입 최종본은 5절 v1 전문 + 7절 CoT 블록(= v2)이며, 채택 모델은 Opus 4.8이다. 향후 다른 모델로 이식할 경우, CoT 블록에 "데이터에 명시된 규칙만 적용하고, 추론으로 규칙을 생성하지 않는다"는 가드 문장을 추가해 Gemini류 규칙 환각을 사전 차단할 것을 권고한다.

## 11. 용어 설명

- **확률적 생성:** 모델이 다음 토큰을 확률 분포에서 표집해 한 글자씩 잇는 방식. 사실 조회가 아니라 그럴듯한 말 생성이라 환각이 발생한다.
- **환각:** 검증 가능한 정답이 있어야 할 질문에서 근거 없이 틀린 정보를 확신하는 답변.
- **Zero-shot / Few-shot / CoT:** 예시 없음 / 입력-출력 예시 제공 / 답 전 단계적 추론 유도.
- **비충실 추론(unfaithful CoT):** 모델의 추론 설명이 실제 판단 근거와 다른 현상. 사후 합리화로 틀린 답을 정당화(Turpin 2023).
- **감사가능성:** 산출물의 각 숫자·주장이 어떤 입력 근거에서 나왔는지 사후에 추적·검증할 수 있는 정도.
- **시스템 프롬프트 / 입력 템플릿 / 유저 프롬프트:** 제품에 고정되는 모델 지시문 / 백엔드가 데이터를 채우는 입력 스키마 / 사용자의 발화.
