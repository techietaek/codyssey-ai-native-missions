# A. 심층 인터뷰

## A1. 비용 제약으로 "무료/저가 모델만" 써야 한다면, 품질을 어떻게 유지할까

전제: 저가·소형 모델은 일반적으로 환각이 잦고 긴 문맥 유지가 약하다. 따라서 **"모델에게 덜 의존하고, 구조와 코드에 더 의존하는"** 방향으로 설계를 옮긴다. 핵심은 *모델이 잘 틀리는 일을 모델에게서 빼앗는 것*이다.

1. **결정적 계산을 코드로 이관(가장 중요):** 전체/완료/지연 수와 진행률은 LLM에게 세게 하지 말고 **백엔드에서 계산해 입력 템플릿에 주입**한다. 본 실습에서 Gemini가 '담당 미정' 행을 빼 헤드라인 합계가 어긋난 사례(완료4+진행6=10≠13)가 바로 "세는 일을 모델에 맡겨서" 생긴 결함이다. 약한 모델일수록 셈을 시키지 말고, LLM은 *문장 서술*만 맡긴다.
2. **검산 레이어(validation)로 출력 거름:** "완료+진행+대기+백로그 = 전체" 같은 체크섬을 코드로 검사해, 불일치하면 출력을 반려하거나 재생성한다. 정책/수치 주장이 입력 근거에 매핑되지 않으면 자동으로 '확인 필요'로 강등한다.
3. **자기일관성(self-consistency) 투표:** 같은 입력으로 N회(예: 3회) 샘플링한 뒤, 구조화 필드(카드 수·지연 목록)는 다수결로 확정한다. 저가 모델의 분산을 비용으로 메우는 방식.
4. **Few-shot·가드레일 강화:** 강한 모델에선 줄여도 되는 예시를, 약한 모델에선 더 촘촘히(거절 사례·경계 케이스 포함) 넣는다. "데이터에 없으면 생성 금지" 같은 가드 문장을 반복·강조한다.
5. **호출 범위 축소:** 한 번에 6개 섹션을 다 만들지 말고, "집계 → 지연 식별 → 서술"을 단계로 쪼개 호출한다. 각 호출의 오류 표면이 작아져 약한 모델도 안정된다(단 호출 수↑ → 비용·지연 트레이드오프).
6. **저신뢰 시 안전한 폴백:** 모델 신뢰도가 낮으면 빈칸을 채우는 대신 '확인 필요'로 비우고 사람에게 넘긴다. 틀린 보고보다 미완 보고가 안전하다.

요약: 무료 모델로 내려갈수록 **"정확성은 코드가, 표현은 모델이"** 로 역할을 재배분하고, 검산·투표·폴백으로 안전망을 두껍게 한다.

## A2. 업무 정책(규정/수치)이 자주 바뀐다면, 환각을 줄이는 운영 전략

본 실습의 최대 교훈이 정확히 이 질문에 답한다: **Gemini v2는 데이터에 없는 '자동 이월 규칙'을 지어냈다.** 정책을 모델의 기억/추론에 맡기는 순간 환각이 시작된다. 따라서 전략의 1원칙은 **"정책을 시스템 프롬프트에 굽지 말고, 주입하라(bake out, inject in)"** 이다.

1. **정책을 단일 출처(SSOT)에서 주입:** 규정·수치는 시스템 프롬프트에 박지 말고, 매 호출 시 **현재 정책 블록을 입력에 주입**(입력 템플릿의 `[정책]` 슬롯)하거나 정책 문서에서 검색(RAG)해 붙인다. 시스템 프롬프트는 "정책은 `[정책]` 블록에 *명시된 것만* 적용한다. 명시 안 된 규칙은 **생성하지 말고** '확인 필요'로 둔다"는 규칙만 갖는다 — 이 한 줄이 Gemini류 규칙 환각을 원천 차단한다.
2. **버전·기준일 스탬프:** 주입 정책에 `정책버전·발효일`을 달고, 출력 다이제스트에도 "적용 정책 vYYYY-MM-DD"를 표기한다. 정책이 낡으면 보고서에서 바로 보인다.
3. **근거 추적(traceability) 강제:** 정책/수치를 언급하는 모든 문장은 주입된 출처 라인에 매핑되어야 하며, 매핑 안 되는 주장은 검산 레이어가 플래그한다.
4. **정책 변경 시 회귀 평가 자동 실행:** 정책이 바뀌면 본 실습의 H1~H5 같은 **고정 환각 평가셋**을 재실행해, 새 정책에서도 모델이 지어내지 않는지 자동 점검한다(평가셋도 정책과 함께 버전 관리).
5. **변동 항목은 코드 계산:** 환율·단가·기한처럼 자주 바뀌는 수치는 LLM에 외우게 하지 말고 계산·조회 결과를 주입한다.

요약: **정책은 데이터처럼 주입하고, 모델에는 "주입된 것만 쓰라"고 명령하며, 변경 시마다 환각 평가셋으로 재검증**한다.

## A3. 10턴+에서 문맥이 끊긴다면, 프롬프트 구조를 어떻게 개선할까

본 실습 v1·v2 6세션은 12턴 문맥 유지를 모두 통과했지만, 끊김이 발생한다면 원인은 보통 "모델이 흐릿한 긴 대화 기록에 의존"하기 때문이다. 해법은 **상태를 대화 기록이 아니라 명시적 객체로 관리**하는 것이다.

1. **'확정 제약' 상태 블록을 매 턴 재주입:** 톤·대상·금지어·표기 형식·카드 변경분을 구조화한 *상태 객체*를 만들어, 매 호출에 다시 붙인다. 모델이 10턴 전 발화를 기억하지 못해도 상태 블록만 보면 된다.
   ```
   [확정 상태]
   - 표기형식: 상태별
   - 톤: 격식 보고 톤
   - 인물표기: 역할명+로마자 이니셜
   - 카드변경: 외부 지도 API=Blocked, 결제 버그 수정 신규(마감 06-27)
   - 보류: velocity·이월규칙·리뷰 표준일 = 확인 필요
   ```
2. **모호 지시 제거:** 본 실습 Q11의 "템플릿 그대로"가 모델마다 담당자별/상태별로 갈렸다. 상태 블록에 `표기형식=상태별`처럼 **형식 플래그를 명시**하면 해석 분기가 사라진다.
3. **체크포인트 요약:** 몇 턴마다 "지금까지 확정된 사항"을 모델이 *재진술*하게 하고, 그 요약을 다음 턴 상태로 승계한다(누적 오차 차단).
4. **보드 상태는 코드가 정본 관리:** 카드 데이터는 대화로 누적하지 말고, 코드가 보유한 구조화 상태를 매 턴 최신본으로 주입한다. 모델은 "현재 상태"만 보고 서술한다.

요약: **긴 대화 기록 대신 명시적 상태 객체를 매 턴 재주입하고, 모호 지시는 플래그로 못박는다.** 문맥 유지를 모델의 기억력이 아니라 입력 설계로 보장한다.

# B. 핵심 원리 자가 설명 (학습 목표)

## B1. LLM은 "확률적 생성" 기반이며, 왜 환각이 발생하는가

LLM은 방대한 텍스트로 "지금까지의 맥락 뒤에 올 다음 토큰의 확률 분포"를 학습한 모델이다. 답을 낼 때는 **데이터베이스를 조회하는 게 아니라, 그 확률 분포에서 토큰을 표집해 한 글자씩 이어 붙인다.** 학습 목표가 "유창하고 그럴듯한 텍스트"이지 "참인 텍스트"가 아니기 때문에, 모델 안에는 진위를 판별하는 별도 장치가 없다. 그래서 데이터에 답이 없을 때도 모델은 *빈칸을 비우기보다 확률적으로 그럴듯한 토큰을 채우려* 한다 — "velocity는 32였습니다" 같은 문장이 *사실이라서*가 아니라 *그럴듯해서* 생성된다. 이것이 환각이다. 따라서 환각을 줄이려면 (a) 시스템 프롬프트로 "모르면 모른다"는 출력을 명시적으로 허용·강제하고, (b) 검증 가능한 사실은 데이터/코드로 주입해 모델이 생성할 필요 자체를 없애야 한다. 본 실습의 안전장치·'확인 필요' 분리가 정확히 (a)다.

## B2. Zero-shot / Few-shot / 단계적 추론 유도(CoT)의 차이와 적용

- **Zero-shot:** 예시 없이 지시만 준다. 빠르고 토큰이 싸지만 형식·거절 패턴이 흔들린다.
- **Few-shot:** "좋은 입력 → 좋은 출력" 예시를 함께 준다. 모델이 형식과 행동(예: 없는 수치엔 '확인 필요')을 *본보기로 흉내* 낸다. 본 과업은 형식 일관성과 환각 거절이 중요해 Few-shot 3개를 채택했고, 그중 1개는 모호 입력→되묻기 사례다.
- **단계적 추론 유도(CoT):** 답 전에 중간 검토 단계를 거치게 한다. 본 과업 v2의 "단계적 검토 규칙"이 이에 해당.

**적용 판단:** 우리 과업은 다단계 수학이 아니라 집계·추출·분류이므로, 문헌(Sprague 2024)대로 CoT의 정확도 이득은 작았고 실제로 v1에서 이미 정답이 나왔다. CoT의 값은 *정확도*가 아니라 *근거 노출(투명성)* 에서 나왔으며, 모델에 따라 이득(Opus·GPT)이거나 역효과(Gemini는 규칙 환각)였다. 즉 **기법 선택은 과업 성격(기호 추론 여부)과 모델 특성에 맞춰야** 하며, "CoT를 켜면 무조건 좋아진다"는 거짓이다. 본 과업의 최적 조합은 **Few-shot(필수) + CoT(투명성 목적, 안정 모델에 한해)** 다.

## B3. "느낌"이 아니라 평가 축·근거로 3종을 비교한다

비교를 재현 가능하게 만든 세 장치: (1) **동일 입력** — 12개 질문 고정 스크립트를 12세션에 똑같이. (2) **평가 축 6개** — 사실 정확성·환각 저항성·형식 준수·문맥 유지·한국어·간결성을 1~5점 루브릭으로. (3) **수치 근거** — 진행률 31%(4÷13), 지연 2건, 환각 직접 질의 통과 같은 *검증 가능한 정답값*으로 채점. 그 결과 "Opus가 좋아 보인다"가 아니라 "4버전 평균 Opus 4.83 / GPT 4.79 / Gemini 4.21, 그중 Gemini는 시스템 프롬프트가 없으면 22/30까지 떨어지고 v2에서도 없는 규칙을 본문에 주입(환각 저항성 5→3)"처럼 **근거를 댈 수 있는 결론**이 나온다. 주관 인상평과의 차이는 *남이 같은 절차로 재현하면 같은 결론에 도달하는가*에 있다.

## B4. 페르소나·시스템 프롬프트가 품질(일관성·안전성·형식 준수)에 미치는 영향

페르소나·시스템 프롬프트는 모델의 **출력 분포를 좁히는 장치**다. 본 실습에서 세 효과가 실측됐다.

- **안전성:** 페르소나의 "정확성 > 친절함" + 안전장치 4번이 Q10 압박("사유 지어내줘")에서 시스템 프롬프트 적용 6세션 전부 날조를 거부하게 했다. 이게 단순한 주장이 아니라는 증거가 **2×2 절제 실험**에 있다 — 시스템 프롬프트를 뺀 v0에서 **Gemini는 같은 압박에 가짜 사유를 실제로 지어냈다**(22/30, 최저). 시스템 프롬프트(v1)가 이를 막아 28/30으로 끌어올렸다(+6). 즉 "정확성 우선"·"압박에 굴복 금지" 같은 한두 줄이 *날조에 취약한 모델의 안전망*으로 작동함이 수치로 확인됐다(모델 비교 보고서 9.4).
- **형식 준수:** `[출력 형식 규칙]`의 고정 템플릿이 6개 섹션을 안정 출력하게 해, 코드 파싱·렌더링이 가능해졌다.
- **일관성:** `[집계 규칙]`이 진행률·지연 정의를 못박았다. 반대로 못박지 않은 '진행'의 범위는 모델마다 흔들렸다(Opus 9 vs Gemini v2 6) — *규칙으로 고정한 것은 일관됐고, 비워둔 것은 흔들렸다*는 대조가 시스템 프롬프트의 힘을 보여준다.

## B5. 생성 결과를 테스트로 검증하고, 반복 개선을 재현 가능하게

재현 가능한 검증·개선 루프를 다음으로 구성했다.

1. **고정 평가셋:** 12개 질문 스크립트 + 환각 5문항(H1~H5) + 각 문항의 기대 정답/Pass·Fail 기준. (문서 02 §8)
2. **객관 채점:** 결정적 정답값(13/4/31%, 지연 2건)과 6축 루브릭으로 채점, 결과를 점수표로 기록. (문서 01 §6)
3. **개선 실험 격리:** v1↔v2의 차이를 CoT 하나로만 두어, 점수 변화가 곧 CoT의 순효과가 되도록 설계. (문서 02 §9)
4. **원본 보존:** 12세션 답변 전문을 `*-log.txt`로 저장 — 누구든 같은 입력·같은 기준으로 재실행·재채점 가능.
5. **문헌 대조:** 결과를 학술 문헌(Sprague·Mind Your Step·Turpin)과 대조해 발견의 타당성을 교차 확인.

이 루프 덕분에 "Opus v2 채택"이라는 결론은 *주장*이 아니라 *재현 가능한 절차의 산출*이 된다. 사실 오류(특히 환각)는 위 (1)(2)로 잡고, 개선은 (3)으로 측정하며, 재현성은 (4)로 보장한다.
